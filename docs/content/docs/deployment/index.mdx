## 1. Architecture Overview

We use **self-hosted GitHub Actions runners** that run directly on the deployment server. This means:

- No SSH connections needed
- Direct Docker access
- Faster deployments (images stay on the same machine)

```
┌─────────────────────────────────────────┐
│           GitHub Actions                │
│    (triggered on push/PR)               │
└───────────────────┬─────────────────────┘
                    │
                    ▼
┌─────────────────────────────────────────┐
│      Self-Hosted Runner (Server)        │
│  ┌─────────┐ ┌─────────┐ ┌─────────┐    │
│  │  Build  │ │  Push   │ │ Deploy  │    │
│  │  Images │ │  Images │ │  Stack  │    │

│       │  (on same server)     │         │

└─────────────────────────────────────────┘
```

## 2. Reverse Proxy (Traefik)

**Traefik** (`infra/traefik/docker-compose.yml`) handles all incoming traffic:

- **Dynamic Discovery**: Connects to Docker socket, auto-discovers containers via labels
- **SSL Certificates**: Automatic Let's Encrypt provisioning and renewal
- **Path-Based Routing**: Routes `example.com/pr-123/` to preview environments
- **Network**: Uses external `traefik` network

---

## 3. Container Orchestration

### Production (`infra/production/docker-compose.prod.yml`)

- Runs `latest` tagged images
- Networks:
  - `traefik`: External network for Traefik routing
  - `app_network`: Internal network for service communication
- Persistent volumes for database

### Preview (`infra/preview/docker-compose.preview.yml`)

- Ephemeral stack per Pull Request
- Isolated by `PREVIEW_NAME` (e.g., `pr-123`)
- Separate PostgreSQL database per preview
- Accessed via `/pr-123/` path prefix

## 4. CI/CD Pipelines

Both pipelines run on **self-hosted runners** with direct Docker access.

### Production Pipeline (`build-push-deploy.yml`)

1. **Build & Push**: Builds Rust backend, Next.js frontend, and docs images
2. **Tagging**: Tags images as `latest` and commit SHA
3. **Deploy**: Pulls images and recreates production stack

### Preview Pipeline (`deploy-preview.yml`)

1. **Trigger**: PR opened, updated, or reopened
2. **Generate Env**: Creates isolated environment for the PR
3. **Deploy**: Starts preview stack at `/pr-{number}/`
4. **Comment**: Posts preview URL as PR comment
5. **Cleanup**: Removes stack when PR is closed

## 5. Automated Testing & Codecov

Before any application image is built or preview is deployed, our CI/CD pipelines trigger automated testing workflows:

- **`backend-tests.yml`**: Runs cargo unit tests, linting (clippy), formatting checks, and uploads the Rust coverage report.
- **`frontend-tests.yml`**:
  1. Runs typechecking and Vitest unit/integration tests.
  2. Uploads the Vitest test coverage to **Codecov**.
  3. Installs Playwright browsers.
  4. Runs the Playwright End-to-End (E2E) tests against the built Next.js server.
  5. Uploads a visual HTML report of the Playwright E2E run as a GitHub Artifact if the job succeeds or fails.

Both pipelines send their coverage metrics (LCOV) directly to Codecov using the `CODECOV_TOKEN` GitHub Action secret. The pipelines are strict; if any test or typecheck fails, the PR preview deployment or Production deployment is blocked automatically.

## 6. Environment Generation

We use composite actions to generate `.env` files from GitHub secrets:

- `.github/actions/generate-prod-env` — Production environment
- `.github/actions/generate-preview-env` — Preview environments

This keeps secrets out of the repository while making them available to Docker Compose.
